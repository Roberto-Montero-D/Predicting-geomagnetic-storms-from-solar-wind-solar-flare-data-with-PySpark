End-to-end machine learning pipeline built with **Apache Spark** to predict geomagnetic storms from solar wind measurements and solar flare activity. The project covers the full data science workflow: raw data ingestion, physics-informed feature engineering, exploratory analysis, and binary classification using **Random Forest** and **Gradient Boosted Trees**.

---

## 🔭 Problem & Physics Context

A **geomagnetic storm** occurs when energy from the solar wind couples strongly with Earth's magnetosphere, disrupting satellite operations, GPS accuracy, power grids, and radio communications. The severity of a storm is measured by the **Kp index** (0–9 scale); values ≥ 5 indicate a significant storm.

Predicting these events in advance requires monitoring the **interplanetary solar wind** as it travels from the Sun to Earth, roughly 1–3 days of travel time. Key physical indicators include:

| Feature | Physical meaning |
|---|---|
| `Bz` (IMF z-component) | When negative, allows energy coupling into the magnetosphere — the most critical storm driver |
| `speed` | Solar wind velocity; high speeds (> 600 km/s) indicate energetic events |
| `electric_field` | Interplanetary electric field; high values correlate with storm onset |
| `plasma_beta` | Ratio of plasma pressure to magnetic pressure; > 1 means solar wind dominates |
| `alfven_mach` | Super-Alfvénic flow compresses Earth's magnetic field strongly |
| `flare_count` | Hourly solar flare count from Fermi GBM; flares are precursors to geomagnetic disturbances |
| `f107` | Solar radio flux at 10.7 cm — proxy for overall solar activity level |
| `sunspots` | Sunspot count follows the 11-year solar cycle; more spots → more flare activity |

## ⚙️ Pipeline Stages

### Stage 1 — ETL

Raw fixed-width text files are parsed with PySpark into typed DataFrames. The pipeline applies:

- **Physical flag filtering**: OMNI encodes missing values as sentinel numbers (e.g. `9999.9`, `99999.0`). Values outside physically meaningful ranges are replaced with `null`.
- **Forward fill**: Missing sensor readings are imputed using the last valid observation (standard practice for time-series telemetry).
- **Cyclical encoding**: `hour` (period 24) and `day of year` (period 366) are encoded as `sin/cos` pairs so the model correctly understands temporal periodicity.
- **GBM aggregation**: Individual flare events are aggregated per calendar hour (`flare_count`, `avg_peak_counts`, `total_energy_proxy`).
- **Left join**: OMNI and GBM are joined on hourly timestamp; hours without flares get zero-filled GBM columns.
- **Lag and rolling features**: Temporal context windows are computed for key predictors (Bz, speed, electric field, kp, flare count).

Processed stages are persisted as **Parquet** — the standard columnar format for Spark-based data pipelines.

### Stage 2 — EDA

All plots are generated programmatically and saved as PNG files to `outputs/eda/`. Key analyses include:

- Distributions and boxplots for all 17 physical features
- Class balance (storm vs. no-storm)
- Pearson correlation matrix across all numeric features
- Time-series plots for key solar wind indicators
- Feature importance charts (generated after training)

### Stage 3 — Training

Two ensemble classifiers are trained and compared:

| Model | Best Val F1 | Test F1 | Precision | Recall |
|---|---|---|---|---|
| Random Forest | 0.8298 | 0.8465 | 0.8922 | 0.8054 |
| **GBTClassifier** | **0.8696** | **0.8671** | **0.9081** | **0.8297** |

**Winner: GBTClassifier** — best on all metrics across both validation and test sets. The val/test gap is minimal (-0.003), confirming the model generalises well without overfitting despite the highly imbalanced dataset (1.26% positive class).

Confusion matrix on the test set (35,064 observations):

| | Predicted: No Storm | Predicted: Storm |
|---|---|---|
| **Actual: No Storm** | 34,173 (TN) | 69 (FP) |
| **Actual: Storm** | 140 (FN) | 682 (TP) |

#### Why not `CrossValidator`?

Spark's built-in `CrossValidator` shuffles data randomly before folding, which **mixes past and future observations**. On time-series data this causes data leakage — the model trains on the future and predicts the past, producing artificially inflated metrics. This project uses a **manual chronological grid search** evaluated on a held-out validation period (2020–2021).

#### Class imbalance

Only **1.26%** of hourly observations correspond to geomagnetic storms (1,995 positive out of 157,776 total). Accuracy is therefore a misleading metric — a model that always predicts "no storm" would score 98.7%. **F1 score** is used as the optimization criterion throughout, balancing precision and recall for the minority class.

## 🚀 Quick Start

### Requirements

- Python 3.10+
- Java 11+ (required by Spark)
- 16 GB RAM minimum recommended (Spark driver uses ~10 GB during training)

### Install

```bash
git clone https://github.com/your-username/geomagstorm.git
cd geomagstorm
pip install -r requirements.txt
```

### Windows users

Spark requires Hadoop's native binaries on Windows. Before running:

1. Download `winutils.exe` and `hadoop.dll` from [cdarlint/winutils](https://github.com/cdarlint/winutils/tree/master/hadoop-3.3.5/bin) and place them in `C:\hadoop\bin\`.
2. Set the environment variable `HADOOP_HOME=C:\hadoop` and add `C:\hadoop\bin` to `PATH`.
3. Set console encoding before running:
```powershell
$env:PYTHONIOENCODING = "utf-8"
spark-submit main.py
```

### Run the full pipeline

```bash
spark-submit main.py
```

### Skip stages for faster iteration

```bash
# ETL + EDA only, skip training
spark-submit main.py --skip-training

# ETL + training only, skip EDA plots
spark-submit main.py --skip-eda

# If processed Parquet already exists, ETL is skipped automatically
spark-submit main.py
```

### Run tests

```bash
pytest
```

Expected output: **14 tests passing**.

## 🔬 Key Engineering Decisions

**Parquet over a relational database** — the original version of this project used a PostgreSQL container to store intermediate datasets. This was replaced with Parquet files for several reasons: no external service dependency, native Spark support, columnar compression, and schema preservation. This is the approach used in real data lake architectures (AWS S3, Databricks, Azure Data Lake).

**Modular Python over Jupyter notebooks** — EDA and model training logic extracted from the original notebook into importable, testable modules. Each module has a single responsibility and is covered by unit tests.

**Physics-informed feature engineering** — lag windows and rolling statistics are not arbitrary. The 1h, 3h, 6h, 12h, 24h lookbacks for Bz reflect the known timescales of solar wind propagation and magnetospheric coupling. The cyclical encoding of hour and day-of-year captures the solar activity cycle without imposing a false linear ordering.
